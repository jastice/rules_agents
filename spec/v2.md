# rules_agents

AI Agent Environment v2: Profile / Runner Model

## 1. Purpose

Build a small Bazel-based system that lets a repository declare a local coding-agent
environment, realize it for a concrete runner, and launch it with one command when the
runner supports launching.

The v2 problem statement is:

- declare portable skills in Bazel from local sources or remote git archives
- build a reusable profile artifact that describes the repo-owned agent environment
- realize that profile for a concrete runner such as `codex` or `claude_code`
- install managed state into the runner's native project-local location
- pass through a declared set of credential environment variables unchanged
- launch the selected runner from the repository root when the runner supports `run`

This is still intentionally narrow. It is not a general AI platform or a global config
manager. It is a repo-owned profile builder and runner launcher.

## 2. Product goal

After a user installs a supported agent client separately, they should be able to:

1. build a profile artifact with Bazel
2. run a concrete runner target that sets up repo-local state
3. launch the selected agent frontend with that state active

Desired user flow:

```bash
bazel build //agent:repo_dev_profile
bazel run //agent:codex_dev_setup
bazel run //agent:codex_dev
```

Expected result:

1. the profile artifact is built deterministically
2. the runner validates required environment variables
3. the runner installs or refreshes generated skills in the native project-local skill root
4. the runner starts the agent from the repository root

## 3. Why this is enough

This shape keeps the repo in control of agent tooling while separating two concerns that
should not be fused:

- what environment the repo wants
- how a concrete client realizes that environment

That is enough to support the current direct CLI runners cleanly and leaves room for future
wrapper-style runners without changing the core build-time model.

## 4. Scope

In scope:

- exactly two implemented runners: `codex` and `claude_code`
- local skills declared from files already in the repo
- remote skills synthesized from git archive downloads
- buildable profile artifacts
- runner manifests derived from profile artifacts
- repo-local managed installation under native skill directories
- `doctor`, `setup`, `run`, and manifest behavior

Out of scope:

- global per-user installation
- user-global agent config synthesis
- secret storage or secret files
- MCP provisioning
- agent binary installation
- provider-specific auth flows
- support beyond `codex` and `claude_code`

## 5. Mental model

The key distinction is:

- `agent_profile` answers: what should be active?
- `agent_runner` answers: how is that profile realized here?

Pipeline:

1. build an `agent_profile`
2. produce a profile artifact
3. run `agent_runner(... )_setup`
4. deploy runner-specific repo-local state
5. run `agent_runner(... )_run` or the runner alias target

This makes profiles build products and runners operational targets.

## 6. Supported runners

### 6.1 Codex

Assumptions:

- skills are filesystem-based directories with `SKILL.md`
- repo-local skills live in `.agents/skills`
- the runner is launched from the repo root

v2 behavior:

- install managed skills into `<repo>/.agents/skills`
- rely on normal repository-root discovery
- do not depend on `CODEX_HOME` for skill installation

### 6.2 Claude Code

Assumptions:

- skills are filesystem-based directories with `SKILL.md`
- repo-local skills live in `.claude/skills`
- the runner is launched from the repo root

v2 behavior:

- install managed skills into `<repo>/.claude/skills`
- rely on normal repository-root discovery
- do not synthesize a global Claude home

## 7. Public API

Use a small surface area.

- `agent_skill`
- `agent_profile`
- `agent_runner`
- `skill_deps`

### 7.1 `agent_skill`

Represents one portable skill bundle.

Suggested shape:

```python
agent_skill(
    name = "repo_helper",
    root = "skills/repo_helper",
    srcs = glob(["skills/repo_helper/**"], exclude_directories = 1),
)
```

Semantics:

- `root` is the logical root of the skill bundle
- the bundle must contain `SKILL.md` at the bundle root
- files under `root` are packaged and preserved relative to that root

Validation requirements:

- fail if no `SKILL.md` exists at the bundle root
- fail if `srcs` escape the declared root

### 7.2 `agent_profile`

Represents one buildable local profile artifact.

Suggested shape:

```python
agent_profile(
    name = "repo_dev_profile",
    skills = [
        ":repo_helper",
    ],
    credential_env = [
        "OPENAI_API_KEY",
    ],
)
```

Semantics:

- `skills` is a list of `agent_skill` targets
- `credential_env` is a list of environment variable names to require and forward unchanged
- `agent_profile` itself is runner-agnostic

Generated targets:

- `:name` — buildable profile artifact
- `:name_manifest` — explicit alias to the profile artifact

### 7.3 `agent_runner`

Represents one concrete runner realization of an `agent_profile`.

Suggested shape:

```python
agent_runner(
    name = "codex_dev",
    runner = "codex",
    profile = ":repo_dev_profile",
)
```

Semantics:

- `runner` is an enum with exactly two implemented values in v2: `codex`, `claude_code`
- `profile` is one `agent_profile` target
- the runner consumes the profile artifact and emits a runner-specific manifest

Generated targets:

- `:name` — default interactive entrypoint, alias to `:name_run`
- `:name_setup` — setup repo-local runner state
- `:name_run` — perform setup, then run the agent frontend
- `:name_doctor` — validate without modifying the workspace
- `:name_manifest` — machine-readable runner manifest

### 7.4 `skill_deps`

Declared in the consumer's `MODULE.bazel`. Resolves remote skill archives by URL and
synthesizes `agent_skill` targets by convention from downloaded content.

## 8. Architecture layers

### 8.1 Layer A: skill packaging

Responsibility:

- package declared skill roots into portable bundle artifacts
- validate bundle shape

Implementation:

- `agent_skill`

### 8.2 Layer B: profile artifact generation

Responsibility:

- aggregate skills and portable profile settings
- render a canonical JSON artifact consumed by runners

Implementation:

- buildable `agent_profile`

Profile artifact schema v2:

```json
{
  "version": 1,
  "profile_name": "repo_dev_profile",
  "credential_env": ["OPENAI_API_KEY"],
  "skills": [
    {
      "skill_id": "main__examples__repo_helper",
      "logical_name": "repo_helper",
      "bundle_runfiles_path": "..."
    }
  ]
}
```

Rules:

- `skill_id` must be deterministic and collision-resistant within a workspace
- no secret values may appear in the profile artifact
- the artifact must contain runfiles-resolvable paths to packaged skill bundles
- the profile artifact must not contain runner-specific install paths

### 8.3 Layer C: runner manifest generation

Responsibility:

- combine a built profile artifact with a concrete runner id
- derive runner-specific install details such as managed directory names

Implementation:

- `agent_runner(... )_manifest`

Runner manifest schema v2:

```json
{
  "version": 1,
  "profile_name": "repo_dev_profile",
  "agent": "codex",
  "credential_env": ["OPENAI_API_KEY"],
  "skills": [
    {
      "skill_id": "main__examples__repo_helper",
      "logical_name": "repo_helper",
      "bundle_runfiles_path": "...",
      "managed_dir_name": "__bazel_agent_env__repo_dev_profile__main__examples__repo_helper"
    }
  ]
}
```

Rules:

- `managed_dir_name` must be deterministic
- `managed_dir_name` must be derived from `profile_name` and `skill_id`
- no secret values may appear in the runner manifest

### 8.4 Layer D: runtime launcher

Responsibility:

- resolve the actual agent binary on the user's machine
- validate credentials
- install generated skills into the native project-local skill root
- launch the agent frontend when requested

Implementation recommendation:

- small Python binary
- use Bazel runfiles support to locate manifest and bundle artifacts

Runtime subcommands:

- `doctor`
- `setup`
- `run`

Rules:

- `run` should call setup first, then exec the agent
- `doctor` must not modify the workspace

### 8.5 Layer E: module extension and remote skill synthesis

Responsibility:

- implement `skill_deps` in `extensions.bzl`
- define the `remote` tag class
- generate `agent_skill` targets by convention from downloaded archives

Convention:

- any directory containing `SKILL.md` at its root becomes one skill target
- target name = directory name
- if the archive root contains `SKILL.md`, it becomes one skill target named after the repo
- nested skill directories inside an already-discovered skill root are not supported in v2

## 9. Install behavior

On every `setup` or `run`:

1. resolve the native skill root for the selected runner
2. create the root if missing
3. for each declared skill:
   - copy the packaged bundle into a temporary sibling directory
   - write an ownership marker inside the temporary directory
   - replace an owned destination atomically when possible
   - fail if the destination exists and is not owned by this tool
4. remove stale managed directories belonging to this profile that are no longer declared,
   but only if they are recorded in the prior cleanup manifest and still carry a matching
   ownership marker
5. remove managed directories belonging to other Bazel-managed profiles for the same agent
   root, but only if they are recorded in those profiles' cleanup manifests and still carry
   matching ownership markers
6. write a profile-local cleanup manifest for diagnostics and future cleanup

Conflict policy:

- the launcher must never delete or overwrite directories it does not own
- ownership requires both:
  - presence in the prior cleanup manifest for the matching profile
  - a matching ownership marker inside the managed directory

## 10. Credential handling

Credential environment variables are declared in `agent_profile`.

Rules:

- the runner requires each declared variable to be present in the parent environment
- the runner forwards declared variables unchanged to the launched child process
- no credential values may be written to disk
- the launcher never writes credential values into either profile or runner manifests

## 11. Launcher behavior

`bazel run //...:runner_name -- <extra args>` should work.

For `run`:

1. load the runner manifest from runfiles
2. resolve workspace root
3. validate credentials
4. perform setup
5. resolve the agent binary from override env, then PATH candidates
6. `exec` the agent from the repository root
7. forward all tail args after `--`

Expected binary overrides:

- Codex: `CODEX_BIN`
- Claude Code: `CLAUDE_CODE_BIN`

The launcher should replace itself with the agent process after setup rather than keeping a
long-lived wrapper process.

## 13. Implementation order

### Step 1: Implement `agent_skill`

Acceptance criteria:

- packaged bundle preserves file layout relative to declared root
- missing `SKILL.md` fails clearly

### Step 2: Implement profile artifact generation

Acceptance criteria:

- `agent_profile` builds a deterministic JSON artifact
- the artifact contains no secrets
- artifact paths resolve correctly via runfiles

### Step 3: Implement runner manifest generation

Acceptance criteria:

- `agent_runner` derives deterministic managed directory names
- one profile artifact can be consumed by multiple runners
- runner manifests remain distinct per runner

### Step 4: Implement runtime launcher with `doctor`

Acceptance criteria:

- `doctor` works with fake agent binaries in tests
- missing binary and missing credentials both produce clear failures

### Step 5: Implement setup logic

Acceptance criteria:

- repeated runs are idempotent
- removed skills are cleaned up
- unmanaged directories are preserved

### Step 6: Implement `run`

Acceptance criteria:

- fake binary sees correct cwd
- fake binary sees forwarded env vars and forwarded args

### Step 7: Implement public `agent_runner`

Acceptance criteria:

- example BUILD file exposes `:runner`, `:runner_setup`, `:runner_run`, `:runner_doctor`,
  and `:runner_manifest`

### Step 8: Implement `skill_deps`

Acceptance criteria:

- remote archives synthesize usable `agent_skill` targets

```python

agent_skill(
    name = "repo_helper",
    root = "skills/repo_helper",
    srcs = glob(["skills/repo_helper/**"], exclude_directories = 1),
)

agent_profile(
    name = "repo_dev_profile",
    skills = [
        ":repo_helper",
    ],
    credential_env = ["OPENAI_API_KEY"],

agent_runner(
    name = "codex_dev",
    runner = "codex",
    profile = ":repo_dev_profile",
)

agent_runner(
    name = "claude_dev",
    runner = "claude_code",
    profile = ":repo_dev_profile",
)
```

Expected user commands:

```bash
bazel build //agent:repo_dev_profile
bazel run //agent:codex_dev_doctor
bazel run //agent:codex_dev_setup
bazel run //agent:codex_dev
bazel run //agent:codex_dev -- --help
```

## 15. Acceptance criteria

The v2 slice is done when all of the following are true:

1. a repo can declare a buildable profile artifact
2. a repo can declare one or more runners from that profile
3. declared skills are packaged by Bazel and installed into the runner's native project-local
   skill root
4. remote skills are resolved from a URL and consumed identically to local skills
5. required env vars are validated and forwarded
6. `bazel run //...:runner_name` launches the selected runner from the repo root
7. `doctor` explains failures clearly
8. rerunning setup and run is safe and deterministic
9. the launcher never writes secret values to disk
10. the launcher never overwrites unmanaged skill directories
## 16. Future extensions intentionally deferred

If this slice works, the most natural follow-ons are:

- project-scoped instructions beyond skills
- wrapper or orchestrator runners beyond direct CLI clients
- MCP declarations on `agent_profile`
- richer profile composition

Do not let any of these distort the narrow shipped model.

## 17. Final guidance

Stay disciplined.

The right v2 is still boring on purpose:

- one small public API
- two implemented runners
- skills only
- env pass-through only
- profile build + runner setup + runner run

Do not invent a universal agent framework. Build the thin layer that proves Bazel can own a
portable, repo-scoped local agent environment and realize it cleanly for concrete runners.
